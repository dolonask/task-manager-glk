API-спецификация «Диспетчерская задач»

ОАО «Государственная лизинговая компания» · Информационная система управления задачами и контроля исполнительской дисциплины · Версия документа 1.0, 13.07.2026

1. Общие положения

Настоящая спецификация описывает REST API серверной части системы «Диспетчерская задач», реализующей ролевую модель, регламент переноса срока задачи и неизменяемый журнал аудита в соответствии с техническим заданием. Клиентская часть — одностраничное веб-приложение — обращается к API исключительно по протоколу HTTPS.

  • Базовый URL: https://dispatcher.glk.kg/api/v1
  • Формат данных: JSON (Content-Type: application/json; charset=utf-8)
  • Формат дат: ISO 8601, даты без времени — YYYY-MM-DD, отметки времени — YYYY-MM-DDTHH:mm:ssZ
  • Кодировка идентификаторов: UUID v4 (строка)
  • Версионирование: версия указывается префиксом пути (/v1); обратно несовместимые изменения выпускаются как /v2

2. Аутентификация и авторизация

Вход в систему осуществляется по учётным данным корпоративного домена glk.kg (Active Directory). Клиент передаёт логин/пароль на сервер аутентификации, который проверяет их через LDAP-биндинг к контроллеру домена и возвращает пару токенов.

2.1. Вход по домену

Метод и путьОписание
POST/auth/loginАутентификация по учётным данным AD (домен glk.kg). Тело: { "username": "ivanov.i", "password": "..." }. Сервер выполняет LDAP-бэкенд аутентификацию и возвращает JWT-токены.
POST/auth/refreshОбновление access-токена по действующему refresh-токену.
POST/auth/logoutИнвалидация текущей сессии (refresh-токена).
GET/auth/meТекущий профиль пользователя, его роль(и), закреплённое СП и список курируемых СП (для члена Правления).
// Ответ POST /auth/login — 200 OK
{
  "accessToken": "eyJhbGciOi...",
  "refreshToken": "8f3d1c2a...",
  "expiresIn": 900,
  "user": {
    "id": "6d1f...", "fullName": "Осмонов Нурлан Асанович",
    "login": "osmonov.n", "role": "board",
    "departmentId": null,
    "curatedDepartmentIds": ["d1", "d2"]
  }
}

Access-токен (JWT, TTL 15 минут) передаётся в заголовке Authorization: Bearer <token> в каждом запросе. Refresh-токен (TTL 12 часов) хранится сервером и используется только для обновления access-токена. Токен содержит sub (id пользователя), role, departmentId, curatedDepartmentIds, iat, exp.

Опционально: усиленная ЭЦП (интеграция с Infodocs) для подтверждения решения по заявке на перенос срока — описана в разделе 6.4 как отдельный флаг запроса.

2.2. Роли и разграничение доступа (RBAC)

Роль (role)Значение
adminАдминистратор — полный доступ, единственная роль, технически изменяющая срок задачи
boardЧлен Правления — постановка задач, согласование переноса срока по курируемым СП
headНачальник СП — декомпозиция на подзадачи, инициирование заявок на перенос
employeeСотрудник СП (опционально) — просмотр и отметка исполнения
observerНаблюдатель (опционально) — доступ только на чтение

Каждый защищённый эндпоинт проверяет роль из JWT и, где применимо, принадлежность пользователя к СП (departmentId) или список курируемых СП (curatedDepartmentIds). Матрица доступа по каждому эндпоинту приведена в соответствующих разделах ниже; сводная матрица — в разделе 8.

3. Общие конвенции API

3.1. Пагинация

Списочные эндпоинты принимают параметры page (по умолчанию 1) и pageSize (по умолчанию 20, максимум 100) и возвращают конверт:

{
  "items": [ /* ... */ ],
  "page": 1, "pageSize": 20, "total": 137, "totalPages": 7
}

3.2. Фильтрация и сортировка

Списочные эндпоинты задач, заявок и журнала аудита принимают query-параметры фильтрации (перечислены в описании каждого эндпоинта) и параметр sort вида -createdAt (минус — по убыванию).

3.3. Формат ошибок

7. Регламент переноса срока

Реализует процедуру из трёх этапов: обоснование (начальник СП) → согласование (курирующий член Правления) → исполнение (администратор). Технически новый срок задачи вносит исключительно администратор методом apply.

Метод и путьОписаниеДоступ
GET/transfer-requestsСписок заявок. Фильтры: status, taskId, departmentId, initiatorId, approverIdпо роли
GET/transfer-requests/{id}Карточка заявкипо роли
POST/tasks/{id}/transfer-requestsСоздание заявки в статусе draft. Тело: proposedDeadline, justification. Отклоняется (409), если по задаче уже есть активная заявкаhead своего СП, admin
PATCH/transfer-requests/{id}/submitdraft → pending. Требует заполненных proposedDeadline и justificationинициатор, admin
PATCH/transfer-requests/{id}/approvepending → approvedboard-куратор СП, admin
PATCH/transfer-requests/{id}/rejectpending → rejected. Тело: decisionReason (обязательно). Срок задачи не меняется, заявка завершаетсяboard-куратор СП, admin
PATCH/transfer-requests/{id}/returnpending → revision. Тело: decisionReason. Возвращается инициатору на доработку и повторную отправкуboard-куратор СП, admin
PATCH/transfer-requests/{id}/applyapproved → applied. Переносит proposedDeadline в task.currentDeadline, пишет запись в журнал аудита с полной трассировкой (инициатор, согласующий, резолюция, исполнитель)admin — единственная роль

Правило одной активной заявки: по одной задаче одновременно допускается не более одной заявки в статусе draft, pending или revision; попытка создать вторую возвращает 409 CONFLICT.

// PATCH /transfer-requests/{id}/reject — тело запроса
{ "decisionReason": "Кампания приурочена к сезонному спросу, перенос недопустим" }

// Опциональное подтверждение усиленной ЭЦП (Infodocs)
{ "decisionReason": "...", "signature": { "provider": "infodocs", "signedPayloadId": "sp_8f21..." } }

11. Журнал аудита

Метод и путьОписаниеДоступ

Эндпоинты изменения или удаления записей аудита в API отсутствуют намеренно — журнал append-only на уровне схемы БД (запись создаётся исключительно триггерами/сервисным слоем при выполнении действий, описанных в разделах 5–7).

12. Нефункциональные требования к реализации API

  • Хранение — реляционная СУБД (PostgreSQL), регулярное резервное копирование.
  • Все мутирующие эндпоинты идемпотентны в рамках заголовка Idempotency-Key (рекомендуется для POST).
  • Ограничение частоты запросов: 100 запросов/мин на пользователя (429 RATE_LIMITED).
  • Локализация сообщений — русский язык.
  • Логи доступа и ошибок хранятся отдельно от бизнес-журнала аудита (раздел 11) и не подменяют его.
